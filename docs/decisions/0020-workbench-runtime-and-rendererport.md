# 0020 — The workbench runtime shell and the `RendererPort` contract

- **Status:** Accepted
- **Date:** 2026-07-07
- **Deciders:** Director + agent
- **Related:** [`../20-architecture.md`](../20-architecture.md) (§2.4 ports, §4 data flow, §5 boundary rules — I2/I4), [`../30-ai-agent-contract.md`](../30-ai-agent-contract.md) (§5 change classification, I2/I3/I4/I8), [`../40-parameterisation.md`](../40-parameterisation.md) (§2.2 `render.*`), [`0019`](./0019-foundation-first-terrain-workbench.md) (the prioritised workbench this realises — esp. item 4, the determinism seam), [`0007`](./0007-wgpu-rendering-framework.md) (the `wgpu`/`winit` renderer adapter, which deferred the camera model and frame contract to "their own ADRs"), [`0017`](./0017-vertex-heightfield-terrain.md) (the height field the renderer reads), [`0004`](./0004-deterministic-core-ports-and-adapters.md) (ports & adapters), [`0002`](./0002-llm-as-strategic-advisor.md) (record-replay rests on I3), [`0015`](./0015-debug-hud-ui-layer.md) (the `egui` overlay that draws inside this adapter), [`0009`](./0009-enforcement-tooling-and-the-gate.md) (the gate); implements issue [#8](https://github.com/eggman0131/providence/issues/8), sets the seam for [#9](https://github.com/eggman0131/providence/issues/9)/[#10](https://github.com/eggman0131/providence/issues/10).

## Context

[ADR 0019](./0019-foundation-first-terrain-workbench.md) made a 3D terrain workbench — a movable-camera `wgpu` view of the height field — a **prioritised, first-class deliverable**, built alongside the terrain core so the Director can judge whether the land feels right. [ADR 0007](./0007-wgpu-rendering-framework.md) chose `wgpu`/`winit` for it and named the renderer a **`RendererPort` adapter**, but deliberately deferred "the camera model … and the terrain **mesh** representation … to later feature/design work or their own ADRs," and it never fixed the **frame contract** — what, concretely, the renderer is handed to draw.

Issue [#8](https://github.com/eggman0131/providence/issues/8) (the first workbench slice: render the static height field as a lit 3D surface, then add an orbit/pan/zoom camera and an on-screen readout) is the first code that needs those answers, and it surfaces a genuinely **new structural shape** the architecture has not yet described. Two forces make this architectural, not a mere implementation detail:

1. **A real-time, window-owned loop inverts the control flow in [`20-architecture.md`](../20-architecture.md) §4.** That data-flow describes a *batch turn loop*: the application steps the sim, then **pushes** a frame to `RendererPort`. But `winit` owns the OS event loop and the main thread — the renderer is called *by* the windowing system on redraw, it does not sit inside an app-driven `for` loop. Introducing a **new port** ([`RendererPort`](../20-architecture.md) §2.4 exists only as a doc-comment stub today) and a new runtime shape is an **architectural change** ([contract §5](../30-ai-agent-contract.md): new port).

2. **Real-time 3D presentation is the classic way determinism leaks in.** A camera carries floats, a window runs at a variable frame rate, and a mouse produces continuous motion — exactly the wall-clock/float/frame-rate coupling that must never reach the integer, pure terrain core ([I3](../30-ai-agent-contract.md)). [ADR 0019](./0019-foundation-first-terrain-workbench.md) item 4 set the *principle* (the camera does its float work "at the edge"; the sim advances on a fixed timestep with discrete recorded commands); this ADR makes the **seam concrete** for the first time so #9 (picking → commands) and #10 (the fixed-timestep loop) build on a fixed foundation rather than inventing one each.

This ADR fixes the **runtime shell and the port contract**. It does **not** re-decide `wgpu`/`winit` ([ADR 0007](./0007-wgpu-rendering-framework.md)), the terrain model ([ADR 0017](./0017-vertex-heightfield-terrain.md)), or the `egui` overlay ([ADR 0015](./0015-debug-hud-ui-layer.md)) — it composes them. Tunable presentation choices (mesh style, lighting, palette, camera limits) remain **config** ([I1](../30-ai-agent-contract.md)) under `render.*`, not decisions for this ADR.

## Decision

We will realise the workbench as a **`wgpu`/`winit` renderer adapter that owns the window loop and draws a derived, read-only terrain snapshot handed to it through a new `RendererPort`**, with the camera held as **adapter-local view state that never crosses the determinism boundary**.

1. **`RendererPort` + a `TerrainFrame` snapshot DTO live in `providence-ports`.** The port is the interface the composition root drives to present terrain. `TerrainFrame` is a **ports-owned, plain, read-only** value — grid dimensions plus a borrow of the row-major heights — a *derived snapshot* of the height field, carrying **no** simulation state and **no** camera/view state. `providence-ports` stays `no_std` and depends on nothing: the DTO is defined in `ports`, so the interface layer never imports `providence-core`, and the renderer adapter never imports `core` either. The application (which may read `core`) builds a `TerrainFrame` from the [`HeightField`](../../crates/core/src/terrain/field.rs) and passes it in — the renderer only ever sees the snapshot.

2. **The renderer adapter owns the `winit` event loop and the `wgpu` surface; the composition root wires and launches it.** We accept `winit`'s control inversion: the adapter runs the loop, and the `providence` composition-root binary (the only crate permitted to name a concrete adapter, [`20-architecture.md`](../20-architecture.md) §5.2) constructs the renderer from config and starts it. Three adapters realise `RendererPort`, exactly as [`20-architecture.md`](../20-architecture.md) §2.4 anticipated: the **on-screen** renderer (windowed `wgpu`/`winit`), a **headless render-to-PNG** capture (the agents-only visual self-check and golden-image basis mandated by [ADR 0007](./0007-wgpu-rendering-framework.md)), and a **no-op test double** (for tests with no GPU).

3. **Camera pose is adapter-local view state and never leaves the adapter.** Orbit/pan/zoom is handled inside the renderer from raw `winit` events; the camera's floats resolve to view/projection matrices **at the edge** and are consumed only by the GPU. The camera is **never** a `TerrainFrame` field, **never** a command, and **never** touches the sim — moving the camera cannot change a single height. This is the concrete form of [ADR 0019](./0019-foundation-first-terrain-workbench.md) item 4's "camera does its float/ray work at the edge," and it is why a real-time 3D view does not endanger [I3](../30-ai-agent-contract.md). (Sim-*affecting* input — picking a vertex to raise/lower — is out of scope here; it arrives in [#9](https://github.com/eggman0131/providence/issues/9) as a discrete, recorded command through `InputPort`, not as camera state.)

4. **Presentation config is projected into a standalone `RenderParams`, off the core-injected `Params`.** `render.*` is authored on the config-loader's `ConfigRoot` and projected by a **separate** mapping into a `RenderParams` type that is **not** a field of [`providence_config::Params`](../../crates/config/src/lib.rs) (the object injected into the deterministic core). `render.*` is non-deterministic, outside-boundary presentation config ([`40-parameterisation.md`](../40-parameterisation.md) §2.2); it flows to the renderer adapter and never reaches the core — the pattern [ADR 0015](./0015-debug-hud-ui-layer.md) set ("render config lands with the presentation adapter"). `render.*` is an **already-registered namespace root**, so this introduces **no new root** and needs no namespace-root decision.

5. **Determinism scope for #8, and the #10 refinement.** In this issue the terrain is **static** — the renderer draws one snapshot while the camera moves — so it mutates nothing and there is no replay surface to protect beyond "the renderer is a pure reader." The contract that makes a **live** sim replayable while a real-time view interpolates it — the fixed-timestep loop, command recording, and render interpolation — is **deferred to [#10](https://github.com/eggman0131/providence/issues/10)**, which **refines** this port (adding how fresh frames are produced on the fixed timestep and interpolated for display); it does not replace it. Designing the seam now, minimally, is what lets #10 be a refinement rather than a redesign.

This is an **architectural** change ([contract §5](../30-ai-agent-contract.md)): it introduces the `RendererPort`, adds the `adapters/renderer` crate to the enforced dependency DAG, and fixes a runtime shape not previously described. The dependency *choices* it relies on are already recorded — `wgpu`/`winit` ([ADR 0007](./0007-wgpu-rendering-framework.md)), `egui`/`egui-wgpu` ([ADR 0015](./0015-debug-hud-ui-layer.md)); the small pinned support crates it adds (`bytemuck`, `pollster`, a PNG writer) are recorded here as adapter-confined utilities in service of that path ([I8](../30-ai-agent-contract.md)).

## Player & experience impact

- **Player / experience:** this is the decision that turns the height field into a **place** — lit, three-dimensional land the Director can orbit, pan, and zoom around. It makes the workbench the real evaluation instrument [ADR 0019](./0019-foundation-first-terrain-workbench.md) promised: "does the stepped land feel right?" ([#11](https://github.com/eggman0131/providence/issues/11)) becomes answerable by *looking*, not by reading a dump. The **camera-is-view-state** rule is what guarantees that exploring the world — turning it, leaning in — is always free and consequence-free: the look never edits the land.
- **Future flexibility:** it fixes the seam every later visual and interactive capability reuses — picking and shaping (#9), the animated fixed-timestep loop (#10), and eventually followers, powers, and the rival rendered on the same surface — **without** committing to any of them now, and **without** letting a real-time view compromise the record-replay reproducibility the LLM-opponent design rests on ([ADR 0002](./0002-llm-as-strategic-advisor.md)). Because the renderer only ever sees a derived snapshot, the terrain model underneath ([ADR 0017](./0017-vertex-heightfield-terrain.md)) stays free to change while it is still being judged.
- **What it forecloses:** it accepts `winit`'s window-owned loop as *the* shell, so the app is structured around an event loop from here on rather than a headless batch driver — a deliberate trade for a visible, movable world early ([ADR 0019](./0019-foundation-first-terrain-workbench.md)).

## Consequences

- **Positive:**
  - The renderer holds **no game state** and imports **no `core`** — it draws a ports-owned `TerrainFrame` — so the boundary (I2/I4) and determinism (I3) hold by construction, not by discipline.
  - The camera-as-view-state seam removes the single biggest way a real-time 3D view could have leaked floats/wall-clock/frame-rate into the core.
  - Three adapters (on-screen, headless-PNG, no-op) fall out of one port, giving agents a real visual self-check loop ([ADR 0007](./0007-wgpu-rendering-framework.md)) and tests a GPU-free double from day one.
  - #9 and #10 extend a fixed seam instead of inventing one, and the `egui` overlay ([ADR 0015](./0015-debug-hud-ui-layer.md)) draws on the surface this adapter already owns — no second GUI/render stack.
- **Negative / trade-offs:**
  - The application is now shaped around a **window-owned event loop**; the batch data-flow in [`20-architecture.md`](../20-architecture.md) §4 is refined for real-time presentation (documented, not silently broken).
  - The `RendererPort` contract is fixed **before** #10's live-sim needs are fully known; mitigated by scoping it to a static snapshot now and explicitly reserving frame-production/interpolation for #10.
  - `wgpu`/`winit` bring a large transitive tree and their own cross-version API churn (managed by pinning + Context7, per [ADR 0007](./0007-wgpu-rendering-framework.md)) — a recurring maintenance touch confined to this one adapter.
- **Enforcement / gate impact:**
  - **Boundary DAG (the real confinement).** `adapters/renderer` (`providence-renderer`) is added to the boundary checker's allowed edges ([`xtask/src/boundary.rs`](../../xtask/src/boundary.rs)) as depending on `providence-ports` + `providence-config` only, and to the `providence` composition-root edge set. This — together with the existing `ZERO_EXTERNAL_DEPS = [core, config, ports]` rule — is what actually keeps `wgpu`/`winit` out of the core. **Note (I6 correction):** [ADR 0007](./0007-wgpu-rendering-framework.md) stated "cargo-deny is scoped so wgpu/winit cannot leak into core"; in this codebase that confinement is enforced by the **boundary checker**, not `deny.toml`. `deny.toml` will still need its `[licenses].allow` list extended for the new transitive licenses, and `wildcards = "deny"` requires the new deps to be version-pinned; a defensive `[[bans.deny]]` naming `wgpu`/`winit` is optional and not the primary mechanism.
  - **The gate stays GPU-free and offline** ([I9](../30-ai-agent-contract.md)). Pure modules (mesh, camera, palette, screen→vertex pick) are unit-tested and run in the gate; GPU-touching code is exercised only through the headless-PNG **capture command** used by `/verify` ([I5](../30-ai-agent-contract.md)) and by non-gating perceptual golden-image checks ([ADR 0007](./0007-wgpu-rendering-framework.md)), **never** as a gated `#[test]`. The renderer crate is **coverage-exempt by default** (unlisted in the [`xtask/src/coverage.rs`](../../xtask/src/coverage.rs) threshold table, as the composition-root binary already is) because its GPU/`winit` glue is not headlessly unit-testable; its pure logic is tested regardless. Tightening this later is a gate-knob change, not an ADR.
  - **`debug-hud` must be built and linted.** The `egui` readout ([ADR 0015](./0015-debug-hud-ui-layer.md), issue #8 Phase 3) compiles only under the `debug-hud` cargo feature, but the gate currently builds **no** features (`clippy --workspace --all-targets`, `llvm-cov --workspace`, both without `--features`). To keep that overlay under the linter, the gate's clippy and coverage steps gain `--features debug-hud` (or `--all-features`). This is an enforcement **strengthening** — it brings feature-gated code under the gate that would otherwise go unchecked — recorded here and cross-referenced from [ADR 0015](./0015-debug-hud-ui-layer.md).
- **Docs to update (this change, I6):** this ADR + [`decisions/README.md`](./README.md) index; [`../20-architecture.md`](../20-architecture.md) (`RendererPort`/`TerrainFrame` now concrete; the renderer adapter crate in §5's layout; §4 data-flow noted as refined for the real-time window loop; camera-as-view-state seam); [`../40-parameterisation.md`](../40-parameterisation.md) (concrete `render.*` keys land with the adapter); [`0007`](./0007-wgpu-rendering-framework.md) (its deferred camera/frame bullets now point here; the cargo-deny wording corrected) and [`0015`](./0015-debug-hud-ui-layer.md) (the `debug-hud` gate-feature note); [`CLAUDE.md`](../../CLAUDE.md) (crate table + `workbench` command). The concrete keys, schema, and `default.toml` entries land **with the code that reads them**, phase by phase ([ADR 0008](./0008-toml-config-format-types-first-schema.md) sequencing).

## Alternatives considered

- **Keep the batch push-a-frame model of [`20-architecture.md`](../20-architecture.md) §4 (app owns the loop, calls `RendererPort.present` each tick).** The right shape for a *headless* engine, but `winit` owns the OS event loop and the main thread — an app-owned `for` loop cannot also service window events without fighting the platform. Rejected for the real-time workbench; the app is structured around the event loop instead, and `present` becomes "here is the current snapshot to draw," called as the loop needs it.
- **Let the renderer import `providence-core` and read `HeightField` directly.** Fewer types — but it couples the adapter to the core's internal representation and invites the renderer to reach for state it should not have. Rejected in favour of a ports-owned `TerrainFrame` snapshot, keeping the renderer a pure reader of a derived value ([ADR 0019](./0019-foundation-first-terrain-workbench.md)).
- **Defer `RendererPort`; have the `providence` binary drive the concrete renderer with no port.** Tempting under "no speculative abstraction" ([contract §4.1](../30-ai-agent-contract.md)) — but [I4](../30-ai-agent-contract.md) requires every side effect behind a port with a test double, and [ADR 0007](./0007-wgpu-rendering-framework.md) already mandates the no-op and headless-PNG adapters. The port is load-bearing now (it *is* the test/verify seam), so it is introduced now, minimally. Rejected.
- **Add `render.*` as a field on the core-injected `Params`.** Simplest wiring, but it drags presentation config across the determinism boundary into the `no_std` core that has no business reading it. Rejected in favour of a separate `RenderParams` projection ([ADR 0015](./0015-debug-hud-ui-layer.md) pattern).
- **Fully specify the fixed-timestep + interpolation loop in this ADR.** More complete, but it decides #10's contract before its needs are on the table and before a live, mutating sim exists to test against. Rejected: this ADR fixes the static-snapshot seam and reserves the live-loop refinement for [#10](https://github.com/eggman0131/providence/issues/10).

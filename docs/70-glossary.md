# 70 — Glossary

> **Status:** Active · **Load this doc for:** quick term lookup. Terms are defined once here; other docs use them without redefining.

## Domain (game) terms

- **Deity / god** — a player (the human, or the rival). Acts on the world only indirectly, via terrain-shaping and powers.
- **Rival deity / opponent** — the AI-controlled god, driven by the LLM strategic advisor. See [`50-llm-opponent.md`](./50-llm-opponent.md).
- **Follower** — a mortal belonging to a deity; lives in a settlement, generates faith, can change allegiance.
- **Settlement** — a cluster of followers built on buildable land; has housing capacity.
- **Terrain vertex** — a grid point (corner) carrying an integer height; the unit of terrain state. Orthogonally-adjacent vertices differ in height by at most `sim.terrain.max_step` (default 1) — the *step invariant*. See [ADR 0017](./decisions/0017-vertex-heightfield-terrain.md).
- **Face** — the square bounded by four adjacent vertices; the derived surface followers build on. **Buildable** = its four corners are equal height (flat), above sea level (dry), and contiguous with other buildable faces. (Height lives on vertices, not on the face.)
- **Step invariant** — the rule that orthogonally-adjacent terrain vertices differ in height by ≤ `sim.terrain.max_step`. Every terrain-mutating operation preserves it; a raise/lower **cascades** to neighbouring vertices to keep it (ADR 0017).
- **Land shaping** — the core verb: raising/lowering terrain vertices (cascading to preserve the step invariant) to make land buildable or to disrupt the rival.
- **Faith / mana** — the single divine resource, generated by followers and spent on shaping and powers. (Used interchangeably; "faith" is the fiction, "mana" the mechanic.)
- **Power** — a data-defined divine intervention from the `content.powers.*` catalogue (e.g. flood, earthquake).
- **Scenario** — a playable map + rules configuration, defined in `content.scenarios.*`.
- **Win / loss condition** — a state-evaluated victory/defeat rule, configured in `sim.winconditions.*`.

## Simulation & time

- **Tick** — the smallest unit of simulation time; the core advances one `step` per tick.
- **Turn** — a higher-level grouping of ticks used for pacing/AI cadence (project usage; where only ticks matter, "tick" is used).
- **Step** — the pure core transition `next = step(state, command, params, rng)`.
- **Seed** — the value that initialises the RNG stream; identical seed + inputs ⇒ identical history (determinism, I3).
- **Determinism boundary** — the line between the pure core (deterministic) and the adapters (may be non-deterministic). See [`20-architecture.md`](./20-architecture.md) §3.
- **Record–replay** — recording resolved decisions so a session can be reproduced without invoking the LLM.

## Architecture terms

- **Deterministic core** — the pure, I/O-free, seeded simulation. Depends only on config data.
- **Application layer** — orchestration that owns a session, applies commands, drives the tick loop, and mediates ports. No game rules.
- **Port** — an interface for a side effect (LLM, renderer, input, persistence, clock/RNG, config, audio, logging). Depended on by name, not by implementation (I4).
- **Adapter** — a concrete implementation of a port; the only layer allowed external libraries/OS/screen/disk.
- **Test double** — a stand-in adapter (mock/scripted/no-op) used in tests so the core/app can be tested without real side effects.
- **Composition root** — the single place where adapters are wired to ports at startup.
- **Observation** — the compact, structured, deterministic snapshot of state passed to the LLM.
- **StrategyDecision** — the schema-validated *intent* the LLM returns; translated to legal commands by the engine.
- **TerrainFrame** — the derived, read-only snapshot of the height field handed to `RendererPort` (grid dimensions + row-major heights; no simulation or camera state). Lets the renderer draw without importing the core. See [ADR 0020](./decisions/0020-workbench-runtime-and-rendererport.md).
- **Workbench** — the movable-camera 3D `wgpu` view of the terrain height field, built so the land can be seen and judged in motion. The camera is adapter-local view state and never touches the sim. See [ADR 0019](./decisions/0019-foundation-first-terrain-workbench.md) / [ADR 0020](./decisions/0020-workbench-runtime-and-rendererport.md).

## Process & config terms

- **Contract** — [`30-ai-agent-contract.md`](./30-ai-agent-contract.md), the governing document.
- **Invariant (I1–I9)** — a non-negotiable rule from the contract §2.
- **The gate** — the one-command local check that runs types, lint, format, boundaries, tests, coverage, and schema validation; the sole definition of "green" (contract §9).
- **ADR (Architecture Decision Record)** — a numbered record of a significant decision; see [`decisions/`](./decisions/).
- **Parameter / config key** — a tunable value expressed as a dot-notation key under a registered namespace root (e.g. `sim.economy.mana.regen_rate`). See [`40-parameterisation.md`](./40-parameterisation.md).
- **Namespace root** — a registered top-level config prefix (`meta`, `sim`, `content`, `ai`, `render`, `input`, `runtime`).
- **Magic number** — a behavioural literal hard-coded in source instead of config; a defect under I1.
- **Content pack** — a bundle of `content.*` definitions (powers, terrain, followers, scenarios) shippable/editable without code.
- **Context Diet** — the rule (in [`README.md`](./README.md)) governing which docs an agent loads for a given task.
- **Bootstrapping paradox** — the problem that enforcement tooling must exist before it can enforce; resolved by the enforcement-first phase (contract §7).
